# Cycle: T001 FP Fix — Chai Property-Style Assertions (#32)

## Context
Chai BDD assertions use property access (not function calls) as test oracles:
```typescript
expect(value).to.be.ok;              // depth 3, no trailing ()
expect(spy).to.have.been.calledThrice; // depth 4
```
Current assertion.scm only matches `call_expression` patterns, so these are missed -> T001 false positive.

4-AI consensus: T001 = "oracle-free" detection. Property-style terminal IS a valid oracle.

## Approach
Explicit terminal allowlist + depth 1-5 patterns in assertion.scm only. No Rust logic changes.

Terminal allowlist (20 properties):
`ok|true|false|null|undefined|exist|exists|empty|NaN|extensible|sealed|frozen|arguments|Arguments|finite|calledOnce|calledTwice|calledThrice|called|notCalled`

Key design decisions:
- `expression_statement` wrapper prevents matching intermediate chains in assignments
- `call_expression` (existing) vs `expression_statement > member_expression` (new) are mutually exclusive AST node types -> no double-counting
- Sinon-Chai properties (`calledOnce` etc.) only matched within expect() chain, not standalone
