## Implementation Notes

### Goal
Phase 8a-4 (#68) の戦略決定: helper delegation による ~979 BLOCK FP をランタイムヒントで解決。
初回実行時に大量BLOCKが出た場合、AI agentが即座に `.exspec.toml` を生成できるよう
`custom_patterns` の存在と書き方を出力に含める。

### Background
- `[assertions] custom_patterns` は技術的に解決済み
- 問題はディスカバラビリティ: ユーザー（特にAI agent）が解決策に辿り着けない
- dogfooding全プロジェクトで T001 BLOCK 10以上のケースが該当

### Design Approach
1. `hints.rs` に `Hint` 構造体と `compute_hints()` を新設（output.rsから責務分離）
2. threshold = 10: 10未満なら `exspec-ignore` で個別対処可能、10以上はhelperパターン未設定が原因
3. `compute_hints()` は unfiltered diagnostics を使用（min-severity フィルタの影響を受けない）
4. `custom_patterns_empty: bool` 引数（YAGNI: パターンリスト自体は不要）
5. Terminal出力: Score行の後にHintセクション追加
6. JSON出力: `hints` 配列追加（空なら省略）
7. SARIF: 変更なし

### Review Rebuttal
- Threshold wording (`10以上` vs test case `10件`) is intentional shorthand, not a spec split. Implementation rule is `>= 10`, and TC-01/TC-02 are the source of truth for the boundary.
- `complexity: standard` is intentionally retained despite 9 listed cases because the work is still Example-only and spans two tightly coupled units (`compute_hints`, formatter output) rather than multiple independent behaviors. The test count reflects output variants, not branching implementation risk.
- `custom_assertion_patterns` and `custom_patterns` serve different audiences. The former is the Rust-side internal field name; the latter is the user-facing TOML key that hints must render. The distinction is deliberate because the feature exists to improve config discoverability, not expose internal API names.
- `custom_patterns_empty: bool` is sufficient for this phase. The hint decision only depends on "user already configured helper patterns or not"; passing the full config would widen the API without adding decision power and can be deferred until a real second hint requires richer context.
